# システム説明書

## 1. 目的

本システムは、配送案件と工事案件を一元管理するための業務管理システムである。

主な目的は、Outlook / Microsoft 365 に届く配送依頼メールの添付 Excel を自動取り込みし、案件情報として登録すること、工事案件を画面から手入力で登録すること、要確認データを人が修正・確認できるようにすること、作業完了報告書や作業チェック表を作成し、必要な情報を売上管理システムへ連携することである。

現時点のリポジトリは、静的 HTML / CSS / JavaScript による画面プロトタイプである。バックエンド、DB、Microsoft Graph API、Excel 解析、AI 解析、PDF / Excel 出力、売上管理システム連携は未実装または想定仕様として扱う。

## 2. 対象業務

### 2.1 配送案件

配送案件は、地域別の共有メールボックスに届くメールを起点に登録する。

- メールサービスは Outlook / Microsoft 365 を想定する。
- メール取得には Microsoft Graph API を利用する想定である。
- メール本文ではなく、添付 Excel ファイルの内容を案件情報として読み取る。
- 1つの Excel ファイルは 1案件として扱う。
- Excel 内の対象シートは「アイザ」シートを想定する。
- 既読メールも取り込み対象に含める。
- 既に処理したメールはメール ID で判定し、重複取り込みを避ける。

配送案件で管理する主な情報は次の通り。

- 案件番号
- 地域
- 差出人
- 受信日時
- メール ID
- 添付ファイル名
- 依頼元情報
- 設置先情報
- 設置商品情報
- 訪問設置日
- 弊社問い合わせ番号
- 注意事項
- 取り込みエラーまたは要確認理由
- 案件ステータス
- 編集履歴

### 2.2 工事案件

工事案件は、メール自動取り込みではなく、画面から手入力で登録する。

工事種別は初期値として次を想定する。

- 弱電
- エアコン
- リフォーム

工事案件で管理する主な情報は次の通り。

- 案件番号
- 地域
- 依頼主
- 依頼主電話番号
- 依頼主住所
- お客様カナ名
- お客様名
- 電話番号
- 住所
- 工事名または工事種別
- 作業予定日
- 作業担当者
- 案件ステータス
- 編集履歴

## 3. 利用者

想定利用者は次の通り。

- 管理者: 全案件の閲覧、編集、設定変更、確認済み操作、再連携、ユーザー管理を行う。
- 一般ユーザー: 担当案件の閲覧、ステータス変更、作業完了報告書、作業チェック表の作成・編集を行う。
- 閲覧ユーザー: 案件と帳票の閲覧のみを行う。

権限により、表示するメニュー、編集可能項目、実行可能な操作を制御する。

- サイン前は保存後も編集できる。
- サイン後は編集不可とする。
- サインを削除すると再編集できる。
- サイン削除履歴は残さない想定である。
- 作業チェック表は売上管理システムへ連携しない。

## 5. 案件ステータス

配送案件のステータスは、手入力で任意変更するのではなく、業務操作の完了に合わせて自動更新する。

配送案件の基本ステータスは次の通り。

- 取り込み済み: ハイアールの配送依頼メール、または添付 Excel の取り込みが完了し、案件として登録された状態。
- 作業確定: 案件内容、訪問設置日、担当者などが確認され、現場作業に進める状態。
- 作業中: 現場担当者が作業開始ボタンを押し、作業が進行中の状態。
- 作業完了: 作業チェック表のお客様サインが保存され、現場作業が完了した状態。
- 完了報告書作成: 作業完了報告書の作成ボタンが押され、報告書作成済みになった状態。
- 連携済み: 売上システムに連携ボタンが押され、売上システムへの連携が完了した状態。
- 要確認: 取り込みエラー、必須項目不足、日付不備、外部連携失敗など、人による確認が必要な状態。

配送案件の通常フローは次の通り。

1. ハイアールのメール取り込みが完了したら、システムが `取り込み済み` にする。
2. 案件確定ボタンを押したら、システムが `作業確定` にする。
3. 作業開始ボタンを押したら、システムが `作業中` にする。
4. チェック表のお客様サインが保存できたら、システムが `作業完了` にする。
5. 完了報告書の作成ボタンを押したら、システムが `完了報告書作成` にする。
6. 売上システムに連携ボタンを押し、連携が成功したら、システムが `連携済み` にする。

配送案件では、ユーザーがステータス文字列を直接選択して変更しない。ステータスは、案件確定、作業開始、サイン保存、完了報告書作成、売上システム連携などの業務ボタン操作に連動して自動更新する。

工事案件のステータスは、配送案件とは別に工事業務の運用に合わせて定義する。配送案件と同じステータスを無理に共用しない。

ステータス変更履歴は、最新状態だけでなく変更者、変更日時、変更前、変更後、変更契機となった操作を記録することが望ましい。

## 6. 外部連携

### 6.1 Microsoft 365 / Outlook

配送案件の自動取り込みに利用する。

想定処理:

1. 地域別の共有メールボックスを監視する。
2. 新着または未処理メールを取得する。
3. 添付 Excel ファイルを取得する。
4. Excel の「アイザ」シートを解析する。
5. 案件情報として登録する。
6. 取り込み結果を記録する。
7. 想定外データは要確認一覧へ登録する。

### 6.2 売上管理システム

見積書、請求書、売上、利益、正式料金などは別システムで扱う想定である。本システムは、案件情報と作業完了報告書情報を API で連携する。

想定連携:

- 案件番号を連携キーにする。
- 作業完了報告書の保存時または編集保存時に連携する。
- 連携成功時は連携状態を更新する。
- 連携失敗時は要確認一覧へ登録する。
- 要確認一覧から再連携できるようにする。

### 6.3 AI 解析

AI は Excel 取り込み時の補助解析に利用する想定である。

想定用途:

- Excel セルの値の正規化
- 商品名、型番、色、数量、備考の抽出補助
- 日付や住所の形式揺れ補正
- 想定外データの検出
- 要確認理由の生成

AI の判断だけで確定登録せず、信頼できない結果は要確認として人が確認する。

## 7. セキュリティ要件

本システムは、お客様の氏名、住所、電話番号、訪問先、作業内容、お客様サイン画像、メール添付 Excel、外部 API 認証情報を扱うため、高い水準のセキュリティを必須要件とする。

### 7.1 保護対象データ

特に厳重に保護する情報は次の通り。

- お客様名
- お客様カナ名
- 電話番号
- 住所
- 訪問設置日または作業予定日
- 設置商品、工事内容、作業内容
- お客様サイン画像
- メール本文、添付 Excel、添付ファイル名
- Microsoft Graph API の認証情報
- 売上管理システム API の認証情報
- ユーザー ID、権限、操作履歴
- 編集前後の値を含む監査ログ

### 7.2 認証

本番環境では、メールアドレスとパスワードだけの簡易認証にしない。

必須要件:

- 管理者アカウントは多要素認証を必須にする。
- 一般ユーザーにも多要素認証を適用できる設計にする。
- パスワードは平文保存しない。
- パスワードを扱う場合は、Argon2id、bcrypt、scrypt などの安全なハッシュ方式を使う。
- セッションには有効期限を設ける。
- 長時間操作がない場合は自動ログアウトする。
- ログイン失敗が連続した場合は、一時ロックまたは追加確認を行う。
- 退職者や委託先変更時に、アカウントを即時無効化できるようにする。

Microsoft Entra ID などの外部 ID 基盤を利用できる場合は、独自パスワード管理より優先する。

### 7.3 認可と権限管理

最小権限の原則を採用する。

基本ロール:

- 管理者: 全案件の閲覧、編集、設定変更、確認済み操作、再連携、ユーザー管理ができる。
- 一般ユーザー: 担当または許可された案件の閲覧、作業報告、チェック表作成、ステータス更新ができる。
- 閲覧ユーザー: 案件と帳票の閲覧のみできる。

制限する操作:

- 管理設定の変更は管理者のみ。
- 確認済み操作は管理者のみ。
- Microsoft Graph API 設定の変更は管理者のみ。
- 売上管理システム連携設定の変更は管理者のみ。
- 一括エクスポートは管理者または明示的に許可されたユーザーのみ。
- お客様サイン後の作業チェック表編集は禁止する。

将来的には、地域別、担当者別、案件種別別のアクセス制御を行えるようにする。

### 7.4 データ暗号化

通信と保存の両方で暗号化を必須とする。

通信:

- すべての画面と API は HTTPS のみ許可する。
- HTTP への平文アクセスは HTTPS へリダイレクトする。
- 外部 API 連携も TLS を必須とする。

保存:

- DB は保存時暗号化を有効にする。
- 添付 Excel、PDF、サイン画像などのファイルも保存時暗号化する。
- バックアップも暗号化する。
- 暗号化鍵はアプリケーションコードやリポジトリに保存しない。
- 暗号化鍵やシークレットは、クラウドの Secret Manager または同等の仕組みで管理する。

### 7.5 個人情報の表示制御

画面上で必要以上に個人情報を表示しない。

推奨:

- 一覧画面では電話番号や住所の全量表示を避ける。
- 詳細画面でのみ必要な個人情報を表示する。
- 閲覧ユーザーには電話番号や住所をマスク表示できるようにする。
- 検索結果や通知文には個人情報を入れすぎない。
- CSV、Excel、PDF 出力は権限で制限する。
- 出力ファイルには必要最小限の項目だけ含める。

### 7.6 添付ファイルと帳票の管理

添付 Excel、作業完了報告書、作業チェック表 PDF は漏えいリスクが高いため、ファイル管理を厳格にする。

必須要件:

- ファイルの直接公開 URL を作らない。
- ファイルアクセス時は必ず認証と権限確認を行う。
- 一時 URL を使う場合は短い有効期限を設定する。
- ファイル名に不要な個人情報を含めない。
- アップロードファイルはウイルススキャンする。
- Excel マクロ付きファイルの扱いを制限または禁止する。
- 帳票ダウンロード履歴を記録する。
- 不要になったファイルは保存期間に従って削除する。

### 7.7 監査ログ

個人情報を扱うため、誰が、いつ、何を見たか、何を変更したかを追跡できるようにする。

記録するイベント:

- ログイン成功、ログイン失敗
- ログアウト
- 案件詳細の閲覧
- 案件情報の作成、編集、削除
- ステータス変更
- 要確認の確認済み操作
- 作業完了報告書の作成、編集、出力、連携
- 作業チェック表の作成、編集、PDF 出力
- サイン取得、サイン削除
- 添付ファイル閲覧、ダウンロード
- 管理設定変更
- ユーザー作成、権限変更、無効化
- 外部 API 連携成功、失敗、再実行

監査ログには、変更前値、変更後値、操作ユーザー、操作日時、対象 ID、IP アドレス、ユーザーエージェントを記録する。ただし、ログ自体に不要な個人情報を過剰に複製しないようにする。

### 7.8 外部 API とシークレット管理

Microsoft Graph API と売上管理システム API の認証情報は厳重に管理する。

必須要件:

- API キー、クライアントシークレット、アクセストークンをソースコードに書かない。
- `.env` や設定ファイルをリポジトリにコミットしない。
- 本番シークレットは Secret Manager などで管理する。
- トークンには必要最小限のスコープだけ付与する。
- Microsoft Graph API は対象メールボックスへのアクセス権を最小化する。
- トークン更新、失効、ローテーション手順を定義する。
- 外部 API 連携失敗時に、認証情報や個人情報をエラーメッセージへ出さない。

### 7.9 入力検証とアプリケーション防御

画面入力、Excel 取り込み、外部 API からのデータはすべて信用しない。

必須要件:

- サーバー側で入力検証を行う。
- XSS 対策として、画面出力時にエスケープする。
- SQL インジェクション対策として、プリペアドステートメントまたは ORM の安全な API を使う。
- CSRF 対策を行う。
- ファイルアップロードでは拡張子だけでなく MIME タイプと内容を検査する。
- Excel 由来の文字列を帳票や画面に出す際もエスケープする。
- API にはレート制限を設ける。
- 管理者向け操作には再認証または確認ダイアログを設ける。

### 7.10 バックアップと保存期間

個人情報は必要な期間だけ保存する。

必須要件:

- DB とファイルのバックアップを暗号化する。
- バックアップから復旧できることを定期的に確認する。
- 案件データ、添付 Excel、帳票 PDF、監査ログの保存期間を定義する。
- 保存期間を過ぎたデータは削除または匿名化する。
- 削除対象、削除方法、削除履歴を管理する。

### 7.11 インシデント対応

漏えい、誤送信、不正アクセス、端末紛失、外部 API 認証情報の流出が起きた場合に備える。

最低限定義する内容:

- 緊急連絡先
- 初動対応手順
- 影響範囲の調査方法
- アカウント停止手順
- API トークン失効手順
- ログ保全手順
- 顧客・関係者への報告判断
- 再発防止策の記録

## 11. 実装時の注意点

- 文字コードは UTF-8 に統一する。
- メール本文ではなく添付 Excel を正とする。
- 1 Excel ファイルを 1案件として扱う。
- Excel の対象シート名は「アイザ」とする。
- 既処理判定にはメール ID を使う。
- AI 解析結果は確定情報ではなく補助情報として扱う。
- 想定外データは要確認一覧に残し、人が確認できるようにする。
- 管理者だけが確認済み操作や設定変更を行えるようにする。
- 重要操作は履歴を残す。
- 作業チェック表のサイン後は編集不可とする。
- 売上管理システム連携失敗は要確認として扱う。
- 顧客情報、サイン画像、添付 Excel、帳票 PDF は暗号化して保存する。
- 個人情報の閲覧、出力、ダウンロード、編集は監査ログに残す。
- API キー、クライアントシークレット、トークンはリポジトリに保存しない。
- 本番環境は HTTPS のみ許可する。

## 13. AI に渡すときの要約

このシステムは、配送案件と工事案件を管理する業務システムである。配送案件は Outlook / Microsoft 365 の地域別共有メールボックスに届く添付 Excel を Microsoft Graph API で取得し、Excel 内の「アイザ」シートから案件情報を読み取って登録する。工事案件は画面から手入力で登録する。取り込み失敗や不確実な AI 解析結果は要確認一覧へ登録し、管理者が修正・確認する。案件詳細を起点に、作業完了報告書と作業チェック表を作成する。作業完了報告書は売上管理システムへ API 連携し、作業チェック表はお客様サイン付き PDF として出力する。現行リポジトリは静的 HTML / CSS / JavaScript の画面プロトタイプであり、バックエンドや外部連携は今後実装する。
