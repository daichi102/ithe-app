# 開発工程表

## 前提

この工程表は、**画面は一応作成済み** の状態から、本番運用できる状態まで開発する場合の想定である。

画面デザインや主要な画面構成は既にあるため、今後の中心作業は以下になる。

- 画面プロトタイプを本番データに接続する
- SQLiteからSupabaseへ移行する
- ログインと権限を入れる
- Microsoft 365 / Outlookから添付Excelを取り込む
- チェック表、完了報告書をPDF化して保存する
- 売上管理システムAPIへ連携する
- 本番公開、バックアップ、運用手順を整える

## 現在の状態

| 項目 | 状態 |
|---|---|
| 画面 | 一応作成済み |
| フロント | HTML / CSS / JavaScriptで構成 |
| 案件データ | `app.js` 内のダミーデータ中心 |
| DB | SQLiteで納品時チェックリストのみ一部保存 |
| 内部API | ローカルPythonサーバーでチェックリスト保存APIのみ実装 |
| 認証 | 未実装 |
| 権限 | 画面上の仮ロールはあるが、本番認証とは未連携 |
| Supabase | 未実装 |
| Microsoft Graph API | 未実装 |
| Excel取込 | 未実装 |
| PDF生成 | 未実装 |
| 売上管理API連携 | 未実装 |
| GPT API | 未実装。第2段階で検討 |

## 開発期間の見積

画面が既にあるため、ゼロから画面を作る場合より短縮できる。  
ただし、本番化に必要なDB、認証、外部API、PDF、運用まわりはこれからなので、一定の期間は必要である。

## 開発期間サマリー

| 範囲 | 平日開発での目安 |
|---|---:|
| 本番最低ライン | 9〜11週間 |
| 標準本番版 | 11〜13週間 |
| GPT見積補助まで含める | 14〜17週間 |

## 本番最低ラインの定義

本番最低ラインでは、以下を完成範囲とする。

| 機能 | 内容 |
|---|---|
| Supabase DB | 案件、チェック表、完了報告、取込ログ、連携ログを保存できる |
| ログイン | Supabase Authでログインできる |
| 権限 | 管理者、現場員の2権限で操作制御できる |
| 案件管理 | ダミーデータではなくDBデータで案件一覧、詳細、ステータスを扱える |
| チェック表 | 入力、保存、再表示、PDF化ができる |
| 完了報告書 | 入力、保存、PDF化、売上管理API連携ができる |
| PDF保存 | チェック表PDF、完了報告書PDFをSupabase Storageに保存できる |
| メール取込 | Microsoft 365 / Outlookから添付Excelを取得できる |
| Excel解析 | 同一フォーマットのExcelから案件登録できる |
| 自動取込 | 手動取込 + 1時間おき自動取込ができる |
| 売上管理API連携 | 完了報告書保存時に送信し、送信済みステータスを表示できる |
| 再試行 | 売上管理API連携失敗時に自動再試行できる |
| 通知 | 再試行後も失敗したら管理者へメール通知できる |
| 操作ログ | 重要操作を記録できる |
| 本番公開 | Vercel + Supabaseで利用できる |

| 作業 | 内容 | 完了条件 |

| API設計 | フロントから呼ぶAPI、Edge Functions、売上管理API連携を整理 | API一覧ができている |

| 帳票項目確認 | 既存画面の入力項目とPDF出力項目を対応付ける | PDFに出す項目が決まっている |
| 売上管理API確認 | 認証方式、送信項目、失敗時仕様を確認する | 不明点リストができている |

### 2週目: Supabase基盤構築

| Storage bucket作成 | PDF保存用bucketを作る | 保存先bucketが作成済み |

| RLS方針設定 | 管理者、現場員のアクセス制御方針を設定 | 最低限のRLSルールができている |

## 第2段階: GPT見積補助

GPT機能は、本番最低ラインとは分けて実装することを推奨する。

理由:

- 料金マスタが売上管理システム側にあり、まず連携仕様の確定が必要。
- GPT単体で正式な見積金額を確定すると、金額誤りのリスクがある。
- 初期本番では、案件管理、チェック表、完了報告、売上連携を安定させる方が優先度が高い。

| 工程 | 期間目安 | 内容 |
|---|---:|---|
| GPT API設計 | 1週間 | 入力項目、出力形式、利用制限、ログ保存を決める |
| 見積候補生成 | 1週間 | 作業内容から見積候補を生成する |
| 料金マスタ連携 | 1〜2週間 | 売上管理システム側の料金マスタと照合する |
| テスト・誤判定対策 | 1週間 | 金額誤り、項目漏れ、異常入力を検証する |

追加期間は **3〜5週間** を見込む。

## 短縮できる部分

画面が既に作成済みのため、以下は短縮できる。

| 短縮対象 | 理由 |
|---|---|
| 画面デザイン | 主要画面が既にあるため、新規デザイン工数を抑えられる |
| 画面遷移 | 既存のメニュー、一覧、詳細、チェック表画面を活用できる |
| 入力項目の洗い出し | 画面上に必要項目がある程度出ている |
| 操作イメージ確認 | プロトタイプを見ながら修正できる |

## まだ時間がかかる部分

画面があっても、以下は本番化で時間がかかる。

| 作業 | 理由 |
|---|---|
| Supabase DB化 | ダミーデータを本番DB構造に移す必要がある |
| 認証・権限 | 顧客情報を扱うため、画面だけでなくDB/API側の制御が必要 |
| Microsoft Graph API | Azure設定、権限、メールボックス監視、添付取得が必要 |
| Excel解析 | 実ファイルの揺れ、重複判定、エラー処理が必要 |
| PDF生成 | 帳票レイアウト、サイン埋め込み、Storage保存が必要 |
| 売上管理API | 相手先仕様に合わせた送信、再試行、ログ管理が必要 |
| 本番運用 | バックアップ、通知、失敗時対応、ユーザー管理が必要 |

## リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 売上管理API仕様が未確定 | 9週目以降が遅れる | 1週目で仕様確認を始める |
| Excel実ファイルに例外が多い | 取込工程が延びる | 早い段階で実ファイルを複数集める |
| PDF帳票の修正が多い | 6週目以降が延びる | 早めにサンプルPDFを出して確認する |
| Supabase権限設計が甘い | 個人情報漏えいリスク | RLSとAPI側チェックを両方入れる |
| 現場スマホ操作が使いづらい | 利用定着しない | 11週目より前にも随時スマホ確認する |
| GPT見積を先に入れたくなる | 本番化が遅れる | GPTは第2段階に分ける |

## 優先順位

| 優先度 | 機能 | 理由 |
|---|---|---|
| 高 | Supabase DB化 | ダミーデータ状態から本番化する土台 |

## 推奨する進め方

1. まず **9〜11週間** で本番最低ラインを完成させる。
2. 画面の大幅作り直しは避け、既存画面を本番データに接続する。
3. 売上管理API仕様とExcel実ファイルは1週目から確認する。
4. PDFは6週目に一度サンプルを出し、早めに帳票確認する。
5. GPT見積補助は第2段階に分ける。
6. 予備期間として12〜13週目を確保し、帳票修正、取込例外、売上API調整に使う。
